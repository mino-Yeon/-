문제 — 동기 방식의 한계

JavaScript는 기본적으로 싱글 스레드 환경에서 동작합니다. 즉, 한 번에 하나의 작업만 처리할 수 있습니다.

서버에서 데이터를 가져오는 작업처럼 시간이 오래 걸리는 작업이 있으면, 그 작업이 끝날 때까지 나머지 모든 작업이 대기해야 합니다.

console.log("작업 시작");

// 서버 요청(2초 후 응답을 주는 서버) — 동기 방식 (세 번째 인자 false)
const xhr = new XMLHttpRequest();
xhr.open("GET", "https://httpbin.org/delay/2", false); // false = 동기
xhr.send(); // 응답이 올 때까지 여기서 멈춤!

console.log(xhr.responseText);    // 응답 도착 후 출력
console.log("나머지 작업 수행"); // 응답이 올 때까지 실행 안 됨
▶ 실행 흐름 — 동기 시뮬레이션

실행 버튼을 누른 뒤 로그가 어떤 순서로 찍히는지 확인하세요.
동기 방식이라 응답이 올 때까지 두 번째·세 번째 줄이 출력되지 않습니다.

동기 방식으로 서버 요청을 하면 응답이 오기 전까지 브라우저가 완전히 멈춥니다. 스크롤도, 클릭도, 화면 렌더링도 불가능합니다.

해결 — 비동기 방식

세 번째 인자를 true로 바꾸고, 응답이 왔을 때 실행할 코드를 콜백 함수로 등록합니다. 요청을 보낸 뒤 응답을 기다리지 않고 다음 코드를 바로 실행합니다.

console.log("작업 시작");

// 서버 요청(2초 후 응답을 주는 서버) — 비동기 방식 (세 번째 인자 true)
const xhr = new XMLHttpRequest();
xhr.open("GET", "https://httpbin.org/delay/2", true); // true = 비동기
xhr.onload = () => {
  // 응답이 왔을 때 실행할 콜백 등록
  console.log(xhr.responseText);
};
xhr.send(); // 요청만 보내고 바로 다음으로 넘어감

console.log("나머지 작업 수행"); // 응답을 기다리지 않고 바로 실행
▶ 실행 흐름 — 비동기 실제 실행

실행 버튼을 누른 뒤 로그 순서를 확인하세요.
"나머지 작업 수행"이 먼저 찍히고, 약 2초 뒤 응답이 도착하면 onload 콜백이 실행됩니다.

응답이 오더라도 현재 실행 중인 작업이 있으면 그게 끝난 후에 콜백이 실행됩니다. 이 순서를 관리하는 것이 이벤트 루프입니다.

비동기의 장점

비동기 방식을 사용하면 시간이 걸리는 작업들을 동시에 처리할 수 있습니다.

같은 요청을 동기로 2번 호출할 때와 비동기로 2번 호출할 때의 총 소요 시간을 비교해보세요.

// 동기 — 요청이 순서대로 처리됨 (총 소요 = 요청1 + 요청2)
syncRequest(); // 첫 번째 응답 올 때까지 대기
syncRequest(); // 그 다음 두 번째 요청 시작

// 비동기 — 요청이 동시에 처리됨 (총 소요 ≈ max(요청1, 요청2))
asyncRequest(); // 요청만 보내고 바로 다음으로
asyncRequest(); // 첫 번째 응답을 기다리지 않고 바로 요청
⏱ 데모 — 동기 2회 vs 비동기 2회 소요 시간 비교

XHR 요청 실습

실제 API에 XHR 요청을 보내고 응답을 확인해봅니다. 각 요청은 서로 독립적으로 실행됩니다.

① 유저 목록 조회

const xhr = new XMLHttpRequest();
xhr.open("GET", "https://dummyjson.com/users", true);
xhr.onload = () => {
  const res = JSON.parse(xhr.responseText);
  console.log("유저들:", res.users);
};
xhr.send();

const xhr = new XMLHttpRequest();
xhr.open("GET", `https://dummyjson.com/posts/user/${userId}`, true);
xhr.onload = () => {
  const res = JSON.parse(xhr.responseText);
  console.log("게시글:", res.posts);
};
xhr.send();
▶ 데모 (userId = 1 고정)

③ 특정 게시글의 댓글 조회

const xhr = new XMLHttpRequest();
xhr.open("GET", `https://dummyjson.com/posts/${postId}/comments`, true);
xhr.onload = () => {
  const res = JSON.parse(xhr.responseText);
  console.log("댓글:", res.comments);
};

세 요청은 각각 독립적으로 실행됩니다. 순서를 보장하려면 응답을 받은 후에 다음 요청을 시작해야 합니다.

콜백 지옥

이번에는 세 요청을 독립적으로 실행하는 것이 아니라, 순서를 맞춰 연결해야 하는 상황입니다.

🎯 목표: 맨 첫 번째 유저가 작성한 첫 번째 게시글의 모든 댓글 정보 조회
즉, users[0].id → posts[0].id → 해당 게시글의 comments 순서로 실행되어야 합니다.

각 요청 결과가 다음 요청의 입력값이 되기 때문에, 응답 콜백 안에 다음 요청을 중첩해야 합니다.

const xhr1 = new XMLHttpRequest();
xhr1.open("GET", "https://dummyjson.com/users", true);
xhr1.onload = () => {
  const res = JSON.parse(xhr1.responseText);
  console.log("유저들:", res.users);

  const xhr2 = new XMLHttpRequest();
  xhr2.open("GET", `https://dummyjson.com/posts/user/${res.users[0].id}`, true);
  xhr2.onload = () => {
    const res2 = JSON.parse(xhr2.responseText);
    console.log("게시글:", res2.posts);

    const xhr3 = new XMLHttpRequest();
    xhr3.open("GET", `https://dummyjson.com/posts/${res2.posts[0].id}/comments`, true);
    xhr3.onload = () => {
      const res3 = JSON.parse(xhr3.responseText);
      console.log("댓글:", res3.comments); // 점점 깊어지는 중첩
    };
    xhr3.send();

  };
  xhr2.send();

};
xhr1.send();
작업이 추가될수록 들여쓰기가 계속 깊어집니다. 이를 콜백 지옥이라고 하며 가독성이 크게 떨어지고 에러 처리도 어렵습니다.

Promise

Promise resolve reject

Promise는 비동기 작업을 처리하기 위한 객체입니다. 콜백 지옥을 없애고, 에러 처리를 쉽게 하며, 비동기 작업을 일관된 방식으로 다룰 수 있게 해줍니다.

3가지 상태

pending
대기 중 — 아직 결과 없음
fulfilled
이행됨 — 성공, resolve 호출
rejected
거부됨 — 실패, reject 호출

XHR 요청을 Promise로 감싸기

콜백 방식의 XHR을 Promise로 감싸면, .then() / .catch() / .finally()로 결과를 처리할 수 있습니다.

먼저 new Promise로 직접 만들고, .then() / .catch() / .finally()로 결과를 처리합니다.

// 2초 후 응답을 주는 서버에 XHR 요청
const promise = new Promise((resolve, reject) => {
  const xhr = new XMLHttpRequest();
  xhr.open("GET", "https://httpbin.org/delay/2", true);
  xhr.onload = () => resolve(JSON.parse(xhr.responseText));
  xhr.onerror = () => reject("네트워크 오류"); // 인터넷 끊김 등 네트워크 자체 실패 시
  xhr.send();
});

promise
  .then(data  => console.log("응답:", data))  // fulfilled 시 실행
  .catch(err  => console.log("실패:", err))   // rejected 시 실행
  .finally(() => console.log("항상 실행")); // 결과 무관하게 실행

같은 요청을 여러 번 써야 한다면, 함수로 감싸서 호출할 때마다 새 Promise를 반환하게 만듭니다.

// Promise를 반환하는 함수로 만들기
function request() {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest();
    xhr.open("GET", "https://httpbin.org/delay/2", true);
    xhr.onload = () => resolve(JSON.parse(xhr.responseText));
    xhr.onerror = () => reject("네트워크 오류");
    xhr.send();
  });
}

// 호출할 때마다 새 Promise 생성
request()
  .then(data  => console.log("응답:", data))
  .catch(err  => console.log("실패:", err))
  .finally(() => console.log("항상 실행"));

Promise 함수화

getUsers getPosts getComments

앞서 실습한 세 XHR 요청을 각각 Promise를 반환하는 함수로 만들면, 재사용하기 쉽고 체이닝도 가능해집니다.

① getUsers()

function getUsers() {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest();
    xhr.open("GET", "https://dummyjson.com/users", true);
    xhr.onload = () => resolve(JSON.parse(xhr.responseText).users);
    xhr.onerror = () => reject("네트워크 오류");
    xhr.send();
  });
}

// 호출 예시
getUsers()
  .then(users => console.log("유저들:", users))
  .catch(err  => console.log("실패:", err));

각 함수가 Promise를 반환하기 때문에 .then()으로 결과를 받을 수 있습니다. 이제 이 함수들을 체이닝으로 연결해봅니다.

Promise 체이닝

.then .catch .finally

.then()에서 Promise를 반환하면 다음 .then()으로 연결할 수 있습니다. 이를 통해 콜백 지옥 없이 순차적인 비동기 처리가 가능합니다.

then 안에서 또 then을 중첩하면 콜백 지옥이 그대로 발생합니다.

getUsers()
  .then(users => {
    getPosts(users[0].id)       // ❌ return 없이 중첩
      .then(posts => {
        getComments(posts[0].id)
          .then(comments => {
            console.log("댓글:", comments);
          });
      });
  });

return으로 Promise를 반환하면 체이닝이 가능합니다. .catch()는 앞선 모든 단계의 에러를 잡습니다.

getUsers()
  .then(users => {
    return getPosts(users[0].id); // ✅ Promise를 return
  })
  .then(posts => {
    return getComments(posts[0].id);
  })
  .then(comments => {
    console.log("댓글:", comments);
  })
  .catch(err => {
    console.log("에러 발생:", err); // 어느 단계에서든 에러 처리
  })
  .finally(() => {
    console.log("작업 완료");
  });

콜백이 단순히 다음 Promise를 반환할 뿐이라면 더 짧게 쓸 수 있습니다.

getUsers()
  .then(users    => getPosts(users[0].id))
  .then(posts    => getComments(posts[0].id))
  .then(comments => console.log("댓글:", comments))
  .catch(err     => console.log("에러 발생:", err))
  .finally(()    => console.log("작업 완료"));
▶ 데모 — Promise 체이닝 (유저 → 게시글 → 댓글)

async / await

async await

Promise 체이닝을 동기 코드처럼 읽히게 작성할 수 있는 문법입니다. async 함수 안에서만 await를 사용할 수 있으며, await는 Promise가 fulfilled 또는 rejected 될 때까지 기다립니다.

async function loadData() {
  const users    = await getUsers();              // Promise가 끝날 때까지 대기
  const posts    = await getPosts(users[0].id);
  const comments = await getComments(posts[0].id);
  console.log("댓글:", comments);
}

loadData(); // 호출 — async 함수는 항상 Promise를 반환

.catch() 대신 try/catch로 에러를 처리합니다.

async function loadData() {
  try {
    const users    = await getUsers();
    const posts    = await getPosts(users[0].id);
    const comments = await getComments(posts[0].id);
    console.log("댓글:", comments);
  } catch (err) {
    console.log("에러 발생:", err); // 어느 단계든 에러 처리
  } finally {
    console.log("작업 완료");
  }
}

loadData();
▶ 데모 — async/await (유저 → 게시글 → 댓글)
▶ 실행 로그
async/await는 Promise를 대체하는 게 아니라 Promise 위에서 동작하는 문법적 편의입니다. 내부적으로는 여전히 Promise입니다.
